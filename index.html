Event System

(WIP explanatation)The EventManager class is a singleton which holds a reference to a number of different events that will trigger actions within the scene. All the classes that use the EventManager will invoke one of its public methods, firing the corresponding event in doing so. As the EventManager is a singleton, any other class in the scene can subscribe to these events. This reduces dependencies between classes, in an effort to keep the software modular and more easily extensible. Any class that needs to listen for these events implements the IEventManagerListener interface, mandating it to implement the subscribeToEvents() method.

Rendering and Initialsing the Models

Our application loads and renders GLB AND GLTF files at runtime. Unity, however, does not natively support these file types. In order to load the models into the application we made use of GLTFUtility, an open source library that supports runtime import of GLB/GLTF files into Unity. It converts the data in the GLB/GLTF files into Unity GameObjects with a MeshFilter and MeshRenderer componenet attached. These components allow Unity to render the model.

Organ.cs

ModelHandler.cs

The model is loaded and initialised within a coroutine. The condition in the yield statement is evaluated each frame, and if it is false, control is passed back to the calling function so that other code can continue to execute while the model is being loaded. Once the condition is true, the remaining code executes and fully initialises the model. The line MaterialAssigner.assignToAllChildren is described more in later sections.

Camera Controls

The user is able to rotate the camera about a pivot position, zoom in and out, and translate the camera left, right, up and down using keybinds. This pivot is initialised to the origin (0,0,0), as this is the position that the model is loaded in. However, this position can be changed by means of the PivotController.

CameraController.cs

The direction of rotation of the camera is determined by the direction vector of the pointer. This rotation is achieved using the .Rotate method, which is a public method of the Transform class provided in the UnityEngine namespace. All Unity GameObjects have a Transform component attached that describes its position and orientation in the scene. The displacement (in the z axis) of the camera from the pivot is determined by the movement of the Scroll wheel. Every frame, the camera's position is set to the pivot about which it is rotating, but is then translated away from the pivot, using the Translate method, by the displacement vector.

The displacement is also used to store the amount displaced in the x and y axes when the keybinds are used.

CameraController.cs

In order to allow models of different sizes to be loaded into the application and viewed in the same way, the initial displacement of the camera from the pivot is determined by modelRadius, the radius of the model (more accurately, the radius of the sphere that bounds the renderer of the model). This radius is also used to determine the scrollspeed attribute of the CameraController class, which means the speed at which the user can zoom in is proportional to the size of the model.

CameraResetter.cs

The position of the camera can also be reset by the user. This is carried out by means of the Lerp function, which performs a Linear Interpolation between two values. For the lerp called on the camera's position, the value returned on any given frame is equivalent to startPos + (TargetPos - startPos) * ratio, where ratio is the ratio of the currentlyElapsed time since the camera started resetting to the totalTravelTime. When the ratio is one, the Camera's position will be equal to the target position and the Lerp is done.

Changing Model Colour and Opacity

To allow the opacity and colour of the segments of loaded models to be changed, our code directly accesses the renderers of the models.

MaterialAssigner.cs

GetComponent""() returns the renderer of the GameObject/model segment passed to the functions. By changing the values of its material, we change how the Unity renderer displays the GameObjects on screen. MaterialAssigner is a static class and houses all methods related to the assignment and manipulation of the materials of loaded models. The functions above in particular are called as part of callback actions to different events triggered by the user (see below).

ModelHandler.cs

We initially encountered some draw ordering issues when attempting to adjust the opacity of the different segments of the model. We resolved these issues by making changes to the shaders our models were using (see Cross Sectional Viewer) and by adjusting the RenderQueue attribute of the material applied to each of the segments of the model. The higher the RenderQueue value, the later an object is drawn to the screen. This is the purpose of the MaterialAssigner.assignToAllChildren function in the loadModel coroutine above

Creating, Storing and Viewing Annotations

Annotations are used for storing text about a particular model in a particular view of that model. To clarify, if the user creates an annotation, the current state of the scene, along with the text they entered, is saved. Selecting this annotation will return the user to that exact view of the model, and display the text in a textbox. This data is stored in the form of an AnnotationData object.

AnnotationData.cs

When a user confirms they want to create an annotation, a new AnnotationData object is instantiated and its fields populated with the relevant data. It is then converted into a JSON string and written to a new file whose name is the same as the title entered by the user. This file is stored in a directory whose name is the same as the model currently being viewed. This directory is initialised as soon as the model is loaded into the program for the first time, in the same directory that the application runs in.

Annotation.cs

The user can select an annotation to view from a dropdown menu. When the user clicks the button to view the dropdown menu, the relevant folder is searched for JSON files. All these files are parsed from JSON back into AnnotationData objects, which are stored in a list. When one is selected, an OnSelectAnnotationevent is fired which holds the relevant AnnotationData object. All subscribers of this event will restore the part of the scene that they are responsible for back to the state encapsulated by the AnnotationData object received (for instance, CameraController restores camera position,rotation and displacement, ModelHandler restores the colours of the model segments, etc).

AnnotationSelector.cs

Cross Sectional Viewer

To implement this feature we made use of surface shaders written in High Level Shader Language. Surface shaders provide a code generation approach to writing shaders that interact with light in Unity. The concept that our cross-sectional view shader employs is similar to the idea of occlusion culling, in which an object will only be rendered by a camera if it is between the near and far clipping planes of the camera. If an object is partially between these two planes, only its polygons between the two planes will be drawn to the screen. Our shader (which we based heavily on this open-source shader, as well as the shaders included in the GLTFUtility library that were already being used) determines which pixels of the model should not be drawn to the screen, based on the position and orientation of a plane that is controlled by the user using sliders.
If the dot product of:

  • The distance between the world space position of a vertex on the mesh of the model and the centre of the plane and
  • The plane's normal
    • is positive, then the current pixel is above the plane, so is discarded.

When the user first presses the cross-sectional view icon, the plane controller is enabled and a different shader is applied to the model. This shader turns the pixels above the plane black.

    The user then controls the plane by means of sliders. One slider controls the y position of the plane, while the other three control the orientation of the plane about the x,y and z axes. This is an example of the type of callback action that is passed to a slider.

    Finally, it's important to note that, although the properties of the shader can be accessed, the shader itself cannot hold a reference to any other object in the scene. The _PlanePosition and _PlaneNormal are arbitrary Vector3 values. Every frame, these values must be updated with the position and normal of the plane being controlled by the user.
