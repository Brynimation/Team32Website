Event System

(WIP explanatation)The EventManager class is a singleton which holds a reference to a number of different events that will trigger actions within the scene. All the classes that use the EventManager will invoke one of its public methods, firing the corresponding event in doing so. As the EventManager is a singleton, any other class in the scene can subscribe to these events. This reduces dependencies between classes, in an effort to keep the software modular and more easily extensible. Any class that needs to listen for these events implements the IEventManagerListener interface, mandating it to implement the subscribeToEvents() method.

Implementation

Rendering and Initialsing the Models

Our application loads and renders GLB AND GLTF files at runtime. Unity, however, does not natively support these file types. In order to load the models into the application we made use of GLTFUtility, an open source library that supports runtime import of GLB/GLTF files into Unity. It converts the data in the GLB/GLTF files into Unity GameObjects with a MeshFilter and MeshRenderer componenet attached. These components allow Unity to render the model.

ModelHandler.cs

The model is loaded and initialised within a coroutine. The condition in the yield statement is evaluated each frame, and if it is false, control is passed back to the calling function so that other code can continue to execute while the model is being loaded. Once the condition is true, the remaining code executes and fully initialises the model.

To allow the opacity and colour of the segments of loaded models to be changed, our code directly accesses the renderers of the models.

GetComponent""() returns the renderer of the GameObject/model segment passed to the functions. Creating, Storing and Viewing Annotations

Annotations are used for storing text about a particular model in a particular view of that model. To clarify, if the user creates an annotation, the current state of the scene, along with the text they entered, is saved. Selecting this annotation will return the user to that exact view of the model, and display the text in a textbox. This data is stored in the form of an AnnotationData object.

AnnotationData.cs

When a user confirms they want to create an annotation, a new AnnotationData object is instantiated and its fields populated with the relevant data. It is then converted into a JSON string and written to a new file whose name is the same as the title entered by the user. This file is stored in a directory whose name is the same as the model currently being viewed. This directory is initialised as soon as the model is loaded into the program for the first time, in the same directory that the application runs in.

Annotation.cs

The user can select an annotation to view from a dropdown menu. When the user clicks the button to view the dropdown menu, the relevant folder is searched for JSON files. All these files are parsed from JSON back into AnnotationData objects, which are stored in a list. When one is selected, an OnSelectAnnotationevent is fired which holds the relevant AnnotationData object. All subscribers of this event will restore the part of the scene that they are responsible for back to the state encapsulated by the AnnotationData object received (for instance, CameraController restores camera position,rotation and displacement, ModelHandler restores the colours of the model segments, etc).

AnnotationSelector.cs

Cross Sectional Viewer

Implementing the cross sectional viewer feature was arguably the most complicated part of the project, as it required the use of surface shaders written in High Level Shader Language. Luckily, Unity does provide skeleton code for the creation of Surface Shaders, so we were by no means starting from scratch. The concept that our shaders employ is similar to occlusion culling, in which an object will only be rendered if it is between the near and far clipping planes of the camera. If an object is partially between these two planes, only its polygons between the two planes will be drawn to the screen. Our shader (which we based heavily on this open-source shader) determines what not to draw to the screen based on the position of plane.

The "surf" function is required for all surface shaders. To determine which polygons of the model are above the plane, the dot product of:

  1. the distance between the world space position of a given pixel on the model and the world space position of the plane
  2. the plane's normal
is calculated. If this value is positive, then the pixel being examined is above the plane and so is discarded using the clip function. This means it is not drawn to the screen by the GPU.
